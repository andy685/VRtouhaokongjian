## 1. 背景与问题

### 1.1 现状

当前头号空间平台对游戏内容实行「构件化」管理：一个游戏由 **元数据**（封面、宣传视频、简介、标签等）和 **10 种资源构件**（客户端 APK、数据包 OBB、PC 客户端、服务器端等）组成。

然而，现有的内容分发机制以 **版本号** 作为唯一的更新判断依据：

```
当前版本号 ≠ 已分发版本号 → 触发「有更新」→ 全量重传
```

这意味着：

- **只改了封面图** → 重新分发整个游戏包（可能是几十 GB）
- **只更新了一个 apk** → 重新下载全部 10 种构件
- 门店带宽被大量浪费，设备更新时间过长，影响营业

### 1.2 核心矛盾

| 场景 | 变更量 | 当前行为 | 合理行为 |
|------|--------|----------|----------|
| 仅封面/素材更新 | KB 级 | 全量重传（可能几十 GB） | 元数据同步，无二进制下载 |
| 仅头显 APK 更新 | 几百 MB | 全量重传所有构件 | 仅下载变更的构件 |
| 增量补丁 | 数十 MB | 下载完整新包 | 下载 patch + 本地合并 |
| 运行架构变化 | 全量 | ✅ 全量重传 | ✅ 全量重传 |

### 1.3 目标

建立 **构件级变更感知** 的更新分发机制，做到：

> **只传输变更的部分，不传输未变的部分。**

## 2. 资源模型

### 2.1 游戏内容拆解

```
游戏 = 元数据层 + 资源构件层
```

#### 元数据层（Metadata Tier）

| 字段 | 类型 | 大小 | 更新影响 |
|------|------|------|----------|
| 游戏封面 `coverUrl` | 图片 URL | ≤5MB | 仅 CDN 缓存刷新 |
| 宣传视频 `videoUrl` | 视频 URL | ≤500MB | CDN 更新 |
| 展位图 `bannerList` | 图片列表 | ≤5MB/张 | CDN 更新 |
| 游戏名称、简介、标签 | 文本 | KB 级 | 数据库更新 |
| 游戏豆消耗、时长限制等 | 配置 | KB 级 | 数据库更新 |
| 版本号 `version` | 文本 | — | 版本记录 |

#### 资源构件层（Artifact Tier）

| 构建角色 | 名称 | 文件类型 | 大小限制 |
|----------|------|----------|----------|
| `headset_client` | 头显客户端 | .apk | ≤2GB |
| `headset_data` | 头显数据包 | .obb/.zip | ≤5GB |
| `pc_client` | PC 游戏客户端 | .exe/.zip | ≤10GB |
| `pc_service` | PC 服务终端 | .exe/.zip | ≤5GB |
| `server_room` | 房间/主控服务 | .exe/.zip | ≤5GB |
| `stream_client` | 串流客户端 | .apk/.exe/.zip | ≤2GB |
| `web_bundle` | WebXR 离线包 | .zip | ≤2GB |
| `media_content` | 媒体内容包 | .mp4/.mov/.zip | ≤10GB |
| `dependency` | 运行依赖/驱动 | .exe/.msi/.zip | ≤2GB |
| `patch` | 补丁/差分包 | .patch/.zip | ≤2GB |

### 2.2 构件指纹

每个资源构件应有独立的内容指纹（如 SHA-256 Hash），用于精确判断是否变更：

```
资源构件 = { 角色 + 文件名 + 文件大小 + SHA256 + 版本标签 }
```

版本号变更只是信号，**不替代**指纹比对。

## 3. 变更分类

### 3.1 三级变更模型

```
Level 0 — 仅元数据变更（Metadata Only）
Level 1 — 部分构件变更（Artifact Partial）
Level 2 — 全量构件变更（Artifact Full）
```

### 3.2 变更判定矩阵

| 变更项 | 级别 | 触发条件 | 分发范围 |
|--------|:----:|----------|----------|
| 封面、展位图、视频 | **L0** | `coverUrl` / `bannerList` / `videoUrl` 变更 | 元数据同步（CDN 刷新） |
| 游戏名称、简介、标签 | **L0** | 文本字段变更 | 元数据同步 |
| 运营配置（豆耗、时长等） | **L0** | 配置字段变更 | 元数据同步 |
| 单个资源构件上传新文件 | **L1** | 该构件 SHA256 变更 | 仅分发该构件 |
| 新增可选构件 | **L1** | 新增构件行 | 仅分发新构件 |
| 上传 patch 差分包 | **L1** | `patch` 构件变更 | 仅分发 patch |
| 运行架构改变 | **L2** | `runtimeArchitecture` 变更 | 按新架构全量分发 |
| 版本号升级 + 多个构件变更 | **L1/L2** | 取决于构件指纹比对结果 | 仅变更的构件 |

### 3.3 判定流程

```
CP 提交更新审核 (?mode=update)
        │
        ▼
┌─ 平台审核 ──────────────────────────────────────────┐
│                                                     │
│  1. 对比元数据字段 → 有变更？→ 标记 L0 变更项        │
│  2. 对比各构件 SHA256 → 有新增/变更/删除？           │
│     ├─ runtimeArchitecture 变了？→ 标记 L2 ✓        │
│     ├─ 仅 patch 构件变更？→ 标记 L1（patch）         │
│     ├─ 部分构件变更？→ 标记 L1（列出变更构件）        │
│     └─ 无构件变更 → 维持当前级别                     │
│                                                     │
│  3. 生成变更清单 (Change Manifest)                  │
│                                                     │
└─────────────────────────────────────────────────────┘
        │
        ▼
   分发执行（见第 4 章）
```

## 4. 分发执行策略

### 4.1 按级别执行

| 级别 | 名称 | 分发行为 | 点播系统行为 | 带宽估算 |
|:----:|------|----------|-------------|----------|
| **L0** | 仅元数据 | 不下发到分发队列；平台直接写 DB + CDN purge | 不做任何下载；下次请求自然拉取新元数据 | ≤5MB |
| **L1** | 部分构件 | 仅分发变更构件；记录构件级分发状态 | 仅下载变更构件；patch 类型触发本地合并 | 根据变更构件大小 |
| **L2** | 全量构件 | 按新架构分发全部必传构件 | 全量下载安装 | 可能数十 GB |

### 4.2 变更清单（Change Manifest）

每次更新审核通过后，生成一份变更清单，随分发任务下发：

```json
{
  "gameId": "G2024001",
  "fromVersion": "v1.2.0",
  "toVersion": "v1.3.0",
  "changeLevel": "L1",
  "metadataChanges": [
    { "field": "coverUrl", "action": "update", "oldHash": "abc123", "newHash": "def456" }
  ],
  "artifactChanges": [
    {
      "role": "headset_client",
      "action": "update",
      "fileName": "game_v1.3.0.apk",
      "fileSize": 268435456,
      "sha256": "e3b0c442...",
      "downloadUrl": "https://cdn.example.com/artifacts/xxx.apk"
    },
    {
      "role": "patch",
      "action": "add",
      "fileName": "v1.2.0_to_v1.3.0.patch",
      "fileSize": 52428800,
      "sha256": "d4c1b553...",
      "downloadUrl": "https://cdn.example.com/patches/xxx.patch",
      "baseVersion": "v1.2.0"
    }
  ]
}
```

### 4.3 点播系统（PC 终端 / 头显）消费逻辑

```
收到 Change Manifest
        │
        ├─ Level = L0？
        │   └─ 仅更新本地元数据缓存，跳过所有下载
        │
        ├─ Level = L1？
        │   ├─ 遍历 artifactChanges
        │   │   ├─ action = add/update → 下载该构件
        │   │   ├─ action = delete → 清理本地构件
        │   │   └─ role = patch → 下载 patch + 本地合并生成新包
        │   └─ metadataChanges 非空 → 更新本地元数据缓存
        │
        └─ Level = L2？
            └─ 全量下载 + 安装
```

### 4.4 静默更新与营业时间保护

| 规则 | 说明 |
|------|------|
| **仅闲时下载** | 在设备参数中开启「仅闲时后台更新」，确保不占用营业时段的带宽 |
| **断点续传** | 所有构件下载支持断点续传，网络中断自动恢复 |
| **原子替换** | 所有构件下载完成且校验通过后才整体切换；任一构件失败则保持旧版本运行 |
| **回滚能力** | 保留上一版本的完整构件清单和本地文件，支持一键回滚 |

## 5. 封面/素材更新的特殊处理

### 5.1 为什么封面更新不应触发游戏重传

封面是**元数据**，不是**资源构件**。它存储在 CDN 上，通过 URL 引用：

```
游戏记录.coverUrl = "https://cdn.example.com/covers/game_001_v2.png"
```

终端（点播系统 / 收银台 / 小程序）通过 HTTP 请求加载封面图。封面更新只需：

1. CP 上传新封面图到 CDN
2. 平台更新 `coverUrl` 字段
3. CDN 对新 URL 缓存预热（或对旧 URL 做 purge）
4. 终端下次渲染游戏列表时自然拉取到新封面

**不需要**：
- ❌ 不触发内容分发任务
- ❌ 不下载任何二进制构件
- ❌ 不让设备进入更新状态

### 5.2 封面更新流程

```
CP 修改封面图 ──提交更新审核──▶ 平台审核（仅素材变更，标记 L0）
                                        │
                                   审核通过
                                        │
                                   ├─ 上传新封面到 CDN
                                   ├─ 更新 DB coverUrl
                                   ├─ CDN purge 旧 URL（可选，控制缓存失效）
                                   └─ 完成（不入分发队列）
```

## 6. 与现有系统的对接

### 6.1 审核页面增强

`/platform/games/review` 审核弹窗需要新增两个区域：

**（1）变更摘要区**（审核更新时显示）

| 内容 | 展示 |
|------|------|
| 版本对比 | v1.2.0 → v1.3.0 |
| 变更级别 | L1（部分构件变更） |
| 元数据变更 | ⚫ 封面图已变更、⚫ 游戏简介已修改 |
| 构件变更 | ⬆ headset_client (256MB)、➕ patch (50MB) |

**（2）构件差异对比**：逐构件展示新旧 SHA256 对比

### 6.2 内容分发页面增强

`/platform/distribute` 的「分发」操作需要增加粒度选项：

| 选项 | 说明 |
|------|------|
| **智能分发（推荐）** | 自动根据 Change Manifest 只分发变更部分 |
| **全量分发** | 强制重新分发全部构件（用于修复异常） |

### 6.3 数据库变更

```sql
-- 游戏版本表新增变更清单字段
ALTER TABLE game_versions ADD COLUMN change_manifest JSON;
ALTER TABLE game_versions ADD COLUMN change_level ENUM('L0','L1','L2');

-- 资源构件表新增指纹字段
ALTER TABLE game_artifacts ADD COLUMN sha256 VARCHAR(64);
ALTER TABLE game_artifacts ADD COLUMN file_size BIGINT;

-- 分发任务表新增构件级状态
ALTER TABLE distribute_tasks ADD COLUMN artifact_status JSON;
-- artifact_status 示例: {"headset_client": "completed", "patch": "downloading"}
```

## 8. FAQ

### Q: 如果 patch 合并失败怎么办？

A: 终端检测到合并失败后，自动降级为完整下载该构件，并向平台上报异常。

### Q: 商户手动触发的「重新分发」怎么处理？

A: 手动重新分发默认走 L2 全量分发，确保覆盖所有异常情况。

### Q: 多个门店对同一游戏的已分发版本不同怎么处理？

A: Change Manifest 携带 `fromVersion` 和 `toVersion`。点播系统上报自身当前版本，平台根据设备当前版本生成差异化的变更清单。如果跨多个版本，累积所有中间变更。

### Q: 封面更新后 CDN 缓存没刷新怎么办？

A: 封面 URL 带版本 hash 后缀，如 `cover_v3_a1b2c3.png`。每次封面变更生成新的 URL，彻底避免缓存问题。

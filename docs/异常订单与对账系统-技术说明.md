## 二、数据模型

```typescript
// useExceptionOrders.ts
interface ExceptionOrder {
  id: number           // 唯一标识（时间戳）
  orderNo: string      // 订单号，如 "VR20260420001"
  store: string        // 店铺名称，如 "我的旗舰店"
  amount: string       // 金额（已格式化），如 "¥88.00"
  type: string         // 异常类型：'金额异常' | '重复支付' | '未到账'
  reason: string       // 异常原因描述
  time: string         // 发生时间
  status: string       // 处理状态：'pending' | 'escalated' | 'rejected' | 'resolved'
  handleRemark: string // 处理备注（驳回时存驳回原因）
  source: string       // 来源：'自动对账' | '手动对账' | '商家提交' | '平台巡检'
}
```

## 三、共享状态机制

### 3.1 useExceptionOrders 组合式函数

文件：`composables/useExceptionOrders.ts`

通过 **Vue 3 模块级 ref** 实现跨组件共享，无需 Pinia/Vuex：

```typescript
// 模块级单例（同一 JS 模块只执行一次）
const exceptionOrders = ref<ExceptionOrder[]>([...])  // 所有页面共享同一份数据
const markDialogVisible = ref(false)                   // 标记弹窗状态
const markTarget = ref({...})                          // 当前标记目标
const markType = ref('')                               // 选中异常类型
const markReason = ref('')                             // 异常说明
```

**导出 API：**

| 方法 | 说明 | 返回值 |
|------|------|--------|
| `openMarkDialog(order)` | 打开标记弹窗 | `false` = 已存在 |
| `confirmMark()` | 确认标记（弹窗确定按钮） | — |
| `isMarked(orderNo)` | 检查是否已标记 | `boolean` |
| `unmarkOrder(orderNo)` | 取消标记 | `boolean` |

### 3.2 MarkExceptionDialog 弹窗组件

通用标记弹窗，被所有订单列表页复用：

```
┌─────────────────────────────┐
│  标记异常订单                  │
│                             │
│  订单号    VR20260420001    │
│  订单金额  ¥88.00           │
│  异常类型  ○ 金额异常        │
│           ○ 重复支付        │
│           ○ 未到账          │
│  异常说明  [选填文本框]       │
│                             │
│        [取消]  [确认标记]     │
└─────────────────────────────┘
```

弹窗通过 `computed` 双向绑定共享 ref，关闭/确认操作同步更新模块级状态。

## 四、标记异常 — 完整流程

### 4.1 涉及页面（共7个）

| 页面 | 角色 | `source` 值 | 金额来源字段 |
|------|------|:-----------:|-------------|
| `shop/CashierOrder.vue` | 商家 | `商家提交` | `row.amount` |
| `shop/OnDemandOrder.vue` | 商家 | `商家提交` | `row.amount` |
| `shop/EditBalanceOrder.vue` | 商家 | `商家提交` | `row.amount` |
| `shop/PointsExchangeOrder.vue` | 商家 | `商家提交` | `'¥0.00'`（无金额体系） |
| `shop/GiftOrder.vue` | 商家 | `商家提交` | `row.giftValue` |
| `shop/ManualDeductOrder.vue` | 商家 | `商家提交` | `row.deductAmount` |
| `platform/OrderFlow.vue` | 平台 | `平台巡检` | `row.amount` 或 `'¥0.00'` |

### 4.2 操作列按钮逻辑

```
         ┌── 是 ← isMarked(orderNo)? ──→ 否 ─────────┐
         │                                             │
    [取消标记]（橙色，可点击）                     [标记异常]（橙色，可点击）
    onClick:                                        onClick:
      unmarkOrder(orderNo)                            if !openMarkDialog({...})
      message.info('已取消标记')                         message.warning('已存在')
                                                      else 弹出 MarkExceptionDialog
```

按钮始终显示，无禁用态。标记和取消均为即时操作，无需二次确认。

### 4.3 confirmMark 内部逻辑

```
confirmMark()
  ↓
type 为空？ → 直接返回（按钮 disabled）
  ↓
exceptionOrders.unshift({
  id: Date.now(),
  orderNo, store, amount, type, reason,
  time: new Date().toLocaleString('zh-CN'),
  status: 'pending',        ← 新增异常统一进入 pending
  handleRemark: '',
  source: 商家提交 | 平台巡检
})
  ↓
关闭弹窗
```

> 新标记的异常插入数组头部，status = `pending`。

## 五、对账中心

### 5.1 商家端对账中心（shop/Reconciliation.vue）

**数据来源：** 直接消费 `useExceptionOrders().exceptionOrders`（共享 ref）

**三个入口产生异常数据：**

| 入口 | 触发方式 | source | 状态 |
|------|---------|--------|------|
| 自动对账 | 点击「自动对账」→ 模拟进度 → 确定性生成差异 | `自动对账` | `pending` |
| 上传对账单 | 上传 xlsx/csv → 15% 概率标记异常 | `手动对账` | `pending` |
| 手动录入 | 填写订单号+渠道金额 → 70% 概率不匹配 | `手动对账` | `pending` |
| 订单页标记 | 各订单列表页「标记异常」按钮 | `商家提交` | `pending` |

**自动对账流程：**

```
点击「自动对账」
  ↓
选择范围（今天/昨天/最近7天）
  ↓
进度动画（4步：拉取订单 → 拉取流水 → 匹配 → 比对）
  ↓
根据当前分钟数确定性生成结果（不是随机，方便演示）
  ↓
exceptionData.push(...)  写入异常
recordData.unshift(...)  写入对账记录
  ↓
展示结果卡片（总订单数 / 对账成功 / 发现异常）
```

### 5.2 平台端对账中心（platform/Reconciliation.vue）

**数据来源：** 独立的本地 `exceptionData` ref（不使用 useExceptionOrders）

**数据构成：** 6 条硬编码 mock 数据，覆盖 pending / rejected 状态，来源混合「商家提交」和「平台巡检」。

### 6.2 商家端处理选项

**弹窗入口：** 异常订单列表 → 点击「处理」/「重新处理」/「查看」

| 订单状态 | 可选操作 | 按钮文字 |
|:--------:|---------|:--------:|
| `pending` | 已解决 / 退款 / 提交平台 / 忽略 | 处理 |
| `escalated` | 已解决 / 退款 / 提交平台 / 忽略 | 处理 |
| `rejected` | 修正后重新提交平台 / 自行解决 | 重新处理 |
| `resolved` | 只读查看 | 查看 |

**处理结果映射：**

| handleResult | 状态变更 | 额外操作 |
|:-----------|---------|---------|
| `resolved` | → `resolved` | — |
| `refund` | → `resolved` | — |
| `escalate` | → `escalated` | `source = '商家提交'` |
| `re_escalate` | → `escalated` | `source = '商家提交'` |
| `ignore` | 从数组删除 | — |

**驳回订单专用 UI：**
```
┌─────────────────────────────────────┐
│  当前状态：[已驳回]（红色标签）        │
│  驳回原因：xxx（红色高亮）             │
│                                     │
│  ⚠ 该订单已被平台驳回，请根据         │
│    驳回原因修正后重新提交。            │
│                                     │
│  处理方式：                          │
│  ○ 修正后重新提交平台                 │
│  ○ 自行解决                         │
└─────────────────────────────────────┘
```

### 6.3 平台端审核选项

**弹窗入口：** 异常订单汇总 → 点击「处理」/「查看」

| handleResult | 状态变更 | 前置校验 |
|:-----------|---------|---------|
| `resolved` | → `resolved` | 无 |
| `refund` | → `resolved` | 无 |
| `reject` | → `rejected` | **备注必填**（驳回原因） |

### 6.4 状态标签映射

| 状态 | 商家端显示 | 平台端显示 | 颜色 |
|:-----|----------|----------|:--:|
| `pending` | 待处理 | 待处理 | 🟡 |
| `escalated` | 平台处理中 | 商家重提 | 🔵 |
| `rejected` | 已驳回 | 已驳回 | 🔴 |
| `resolved` | 已处理 | 已处理 | 🟢 |

## 八、与收银系统（cashier-ui）的关系

**结论：无关系，不需要对齐。**

- cashier-ui 是独立项目（Element Plus），不共享 admin-dashboard 代码
- 收银台职责是快速收银，异常标记属于管理后台的对账环节
- 异常标记功能仅在 admin-dashboard 中实现即可

# 头号空间 - 支付逻辑说明

> 版本：v1.1  
> 日期：2026-06-11  
> 适用范围：商家收银系统、会员扣费、PC点播系统、小程序会员支付、订单与退款服务

> **v1.1 更新**：新增加 §11"扫码支付流程"章节，涵盖扫码枪识别、双弹窗架构、两层校验机制、手动输入备选、错误提示规范。

---

## 1. 文档目标

本文档定义头号空间各终端的支付方式、会员资产自动抵扣规则、补差规则、退款规则、小票与结算后提示口径。

所有规则均围绕一个原则：**涉及钱和会员资产的行为必须可预览、可确认、可追溯、可对账**。

---

## 2. 关键定义

| 名称 | 定义 | 账务属性 |
|------|------|----------|
| 预存款 | 会员账户内的人民币余额，充值后进入会员账户 | 会员负债，按人民币金额记账 |
| 游戏币 | 会员账户内的消费资产，汇率与预存款为 1:1，多数来自充值赠送 | 会员资产，按数量记账，同时可折算人民币金额 |
| 游戏豆 | 商家或平台侧用于启动/消耗游戏的 B 端资源 | B 端运营资源，不是用户支付方式 |
| 外部支付 | 现金、微信支付、支付宝支付等实际收款渠道 | 实收资金 |
| 优惠券/活动优惠 | 抵减订单应付金额的优惠权益 | 冲减应收，不是支付方式 |
| 补差 | 会员资产不足时，剩余金额由外部支付补足 | 混合支付的一部分 |

> 注意：`游戏币` 和 `游戏豆` 是两套不同资产。游戏币面向会员，游戏豆面向商家/平台计费。

---

## 3. 支付方式范围

未来商品或项目建议增加"可用支付范围"配置。

| 配置项 | 说明 |
|------|------|
| 仅外部支付 | 只能用现金、微信、支付宝等外部方式支付 |
| 可预存款 | 可使用会员预存款自动抵扣 |
| 可游戏币 | 可使用游戏币自动抵扣 |
| 仅游戏币 | 只能使用游戏币，游戏币不足时不可外部补差 |
| 可混合支付 | 可按系统规则使用会员资产，不足部分走外部补差 |

默认未配置时，建议按普通消费处理：支持优惠、预存款、游戏币、外部补差。

**微信/支付宝支付需扫码枪**：
- 收银员选择微信/支付宝支付后，必须使用扫码枪扫描玩家付款码。
- 系统对付款码进行格式校验 + 后端有效性验证，验证通过后方可完成支付。
- 扫码枪故障时可切换为手动输入模式。
- 现金和会员资产抵扣无需扫码，直接确认即可。

## 4. 自动抵扣规则

### 4.1 默认优先级

默认消耗顺序：

```
优惠券/活动优惠 → 预存款 → 游戏币 → 外部支付补差
```

说明：

- 优惠先抵扣订单金额，再计算会员资产消耗。
- 预存款默认优先于游戏币。
- 游戏币不作为收银员手动选择的支付按钮出现。
- 游戏币由系统根据商品支付范围和商户优先级自动计算是否消耗。
- 商户可以自定义会员资产消耗优先级，例如 `游戏币 → 预存款`，但不能突破商品支付范围限制。

### 4.2 游戏币规则

| 规则 | 说明 |
|------|------|
| 汇率 | 游戏币与预存款按 1:1 抵扣 |
| 是否可手动选择 | 不允许主动选择，由系统自动决策 |
| 是否可用于玩游戏 | 必须支持，游戏币可以作为游戏体验类消费资产 |
| 是否可与优惠券叠加 | 支持，先优惠后抵扣 |
| 是否可与预存款混用 | 支持，由商户优先级决定顺序 |
| 是否可与外部支付混用 | 普通混合支付支持，`仅游戏币` 商品不支持补差 |
| 有效期消耗顺序 | 优先消耗最快过期的游戏币 |

游戏币来源：

```
充值赠送
活动赠送
后台发放
```

游戏币目前不设置售卖型来源。若未来新增直接购买游戏币的业务，应作为新的充值类商品单独评审，并继续遵守"充值只能用外部支付"的规则。

### 4.3 充值规则

充值储值只能使用外部支付：

```
现金 / 微信支付 / 支付宝支付
```

充值不允许使用：

```
预存款 / 游戏币 / 积分
```

原因：充值是向会员账户增加资产，不能用账户内已有资产再次充值。

## 5. 终端支付逻辑

### 5.1 商家收银系统

| 场景 | 支付逻辑 |
|------|----------|
| 散客普通消费 | 外部支付 |
| 会员普通消费 | 按商品范围自动使用预存款/游戏币，不足时外部补差 |
| 会员购买游戏体验 | 支持预存款和游戏币，按商户优先级自动抵扣 |
| 充值储值 | 仅外部支付 |
| 仅游戏币商品 | 仅扣游戏币，不允许外部补差 |
| **微信/支付宝补差** | **必须经过扫码枪扫码 → 格式校验 → 后端验证 → 确认支付** |
| **现金补差** | 直接确认，无需扫码 |
| **会员资产全额抵扣** | 直接确认扣款，无需选择支付方式 |

### 5.2 会员扣费页面

会员扣费页面定位为"会员账户资产扣减"。

建议规则：

- 必须先选择会员。
- 根据商品支付范围自动计算扣预存款或游戏币。
- 如果商品允许混合支付，可展示外部补差入口。
- 如果商品是 `仅游戏币`，游戏币不足时禁止扣费。
- 扣费前必须展示本次资产消耗明细。

### 5.3 PC点播系统

PC点播系统建议区分以下订单来源：

| 订单类型 | 支付逻辑 |
|------|----------|
| 小程序内会员支付 | 用户确认后按会员资产优先级抵扣，不足时微信/支付宝补差 |
| 直接支付-支付宝 | 不消耗会员资产，仅支付宝支付 |
| 直接支付-微信 | 不消耗会员资产，仅微信支付 |
| 会员点播 | 按会员资产优先级自动抵扣，可补差 |
| 直接点播 | 外部支付，不走会员资产 |

### 5.4 小程序会员支付

小程序端必须让用户确认资产消耗明细：

```
本单将使用：
预存款 ¥60.00
游戏币 20
微信补差 ¥10.00
```

用户确认后才发起扣款或补差支付。

## 6. 结算前预览

结算弹窗或结算面板必须展示系统自动计算结果。

示例：

```
商品合计        ¥100.00
优惠券          -¥10.00
预存款抵扣      -¥60.00
游戏币抵扣      -20
仍需支付        ¥10.00
```

如果无需补差：

```
商品合计        ¥80.00
预存款抵扣      -¥50.00
游戏币抵扣      -30
仍需支付        ¥0.00
```

按钮文案建议：

| 场景 | 主按钮文案 |
|------|------------|
| 需要外部补差（微信/支付宝） | 扫码支付（微信）¥xx.xx / 扫码支付（支付宝）¥xx.xx |
| 需要外部补差（现金） | 确认并支付 ¥xx.xx |
| 全部会员资产抵扣 | 确认扣款 |
| 仅游戏币且余额充足 | 确认兑换 |
| 仅游戏币且余额不足 | 游戏币不足（禁用） |

## 7. 退款规则

退款原则：**所有支付与资产消耗均原路返回**。

| 原支付/抵扣方式 | 退款方式 |
|------|----------|
| 现金 | 线下现金退回，并记录退款操作 |
| 微信支付 | 原微信渠道退款 |
| 支付宝支付 | 原支付宝渠道退款 |
| 预存款 | 退回会员预存款 |
| 游戏币 | 退回会员游戏币 |
| 优惠券 | 按券规则恢复或不恢复，需记录原因 |

全额退款：

```
按原订单支付明细全部原路退回
```

部分退款：

```
优先按被退商品对应的支付构成原路退回
无法精确匹配商品时，按整单支付比例原路退回
```

退款必须保留原始支付流水，并新增退款/冲正流水，不能直接改写原流水。

## 8. 成功提示与小票

### 8.1 支付成功提示

结算成功后必须展示"支付拆分"和"会员资产变化"。

示例：

```
支付成功

支付明细
优惠券：-¥10.00
预存款抵扣：¥60.00
游戏币抵扣：20
微信支付：¥10.00

会员资产
预存款：¥300.00 → ¥240.00
游戏币：2704 → 2684
```

### 8.2 小票字段

小票建议包含：

- 订单号
- 终端来源
- 门店名称
- 收银员/设备
- 商品明细
- 优惠明细
- 预存款抵扣
- 游戏币抵扣
- 外部支付补差
- 实收金额
- 会员剩余预存款
- 会员剩余游戏币
- 退款状态或原路退款说明

小票上的会员资产金额必须与订单支付明细和会员资产流水一致。

## 9. 风控与异常

| 场景 | 处理规则 |
|------|----------|
| 会员资产扣减失败 | 订单不应完成，外部支付不应继续 |
| 外部补差支付失败 | 已冻结或预扣的会员资产需释放或冲正 |
| 游戏币过期 | 不可被抵扣，结算预览中应说明不可用原因 |
| 优惠券失效 | 自动取消选择，重新计算应付 |
| 重复提交 | 使用订单号/支付会话号做幂等 |
| 资产不足 | 显示差额和可选补差方式 |

## 11. 扫码支付流程（v1.1 新增）

收银员选择微信支付或支付宝支付时，必须通过扫码枪扫描玩家付款码完成支付。系统采用**双弹窗架构**和**两层校验机制**确保支付安全。

### 11.1 双弹窗架构

收银系统包含两个支付相关弹窗：

| 弹窗 | 触发条件 | 功能 |
|------|---------|------|
| **结算弹窗** | 收银员点击"去结算" | 展示订单明细、会员资产抵扣、支付方式选择 |
| **扫码弹窗** | 选择微信/支付宝后点击确认 | 扫描玩家付款码，两层校验，手动输入备选 |

现金和会员资产全额抵扣不弹出扫码弹窗，直接在结算弹窗确认即可。

### 11.2 支付分流逻辑

```
收银员点击结算确认
    ├── 现金支付 → doConfirm() → 支付成功
    ├── 会员资产全额抵扣 → doConfirm() → 扣款成功
    └── 微信/支付宝 → openScanner() → 扫码弹窗
```

### 11.3 扫码弹窗状态机

扫码弹窗有 4 个状态：

| 状态 | 说明 | 视觉表现 |
|------|------|---------|
| `scanning` | 等待扫码输入 | 深色扫码框 + 四角白色边框 + 呼吸动画图标 + 激光扫描线动画 |
| `verifying` | 正在校验付款码有效性 | 扫码框变暗金色 + 四角黄色边框 + 旋转加载动画 + "验证付款码中..." |
| `success` | 付款码有效 | 扫码框变深绿色 + 四角绿色边框 + ✓图标 + "扫码成功" |
| `error` | 付款码无效 | 扫码框变深红色 + 四角红色边框 + ✕图标 + 具体错误原因 |

### 11.4 两层校验机制

#### 第 1 层：格式校验（即时反馈）

扫码枪输入完成（80ms 无新输入）或回车后，立即进行格式校验：

| 校验项 | 规则 | 错误提示 |
|--------|------|---------|
| 位数不足 | 付款码长度 < 15 | `付款码位数不足（当前X位，至少15位）` |
| 非数字字符 | 付款码含非数字字符 | `付款码只能包含数字，请重新扫描` |
| 位数超长 | 付款码长度 > 20 | `付款码位数超长（当前X位，最多20位）` |

格式校验失败后，错误提示停留 2.5 秒，自动恢复为扫描状态，可重新输入。

#### 第 2 层：后端验证（~1.2s 延迟）

格式校验通过后，系统进入 `verifying` 状态，模拟调用后端 API 验证付款码是否真实有效：

| 检测项 | 规则 | 说明 |
|--------|------|------|
| 连续相同数字 | ≥8 个相同数字（如 `111111111111111111`） | 明显虚假的付款码 |
| 连续递增/递减 | ≥6 位连续递增或递减（如 `123456789012345678`、`987654321098765432`） | 简单输入序列 |

> **真实接入指引**：生产环境中将 `simulateBackendVerification()` 替换为实际的 `POST /api/verify-payment-code` 调用，后端对接支付宝/微信支付服务商接口进行付款码验证。

验证失败后错误提示停留 3 秒，同时手动输入框同步变红提示。

### 11.5 扫码枪输入处理

| 特性 | 说明 |
|------|------|
| **输入方式** | 隐藏的 `<input>` 自动聚焦，接收扫码枪键盘事件 |
| **防抖识别** | 80ms 内无新输入即判定扫码完成，自动触发识别 |
| **回车识别** | 部分扫码枪以回车结束，回车键直接触发识别 |
| **手动输入备选** | 扫码枪故障时，点击"手动输入付款码"切换为文本框输入 |

### 11.6 手动输入

扫码枪故障或特殊情况下，收银员可点击"手动输入付款码"切换为文本框输入模式：

| 交互元素 | 说明 |
|---------|------|
| **输入框** | 支持输入 15-20 位数字，`SF Mono` 等宽字体 |
| **格式提示** | 无错误时显示"支持支付宝/微信 15~20 位数字付款码" |
| **行内错误提示** | 红色边框 + 红色背景错误卡片，精准提示具体错误 |
| **错误自动清除** | 用户重新输入即清除错误状态 |

手动输入同样经过两层校验，与扫码枪输入完全一致。

### 11.7 错误提示规范

| 场景 | 扫码框提示 | 手动输入框提示 |
|------|-----------|--------------|
| 格式错误（位数/非数字） | 扫码框变红 + 错误原因 | 输入框红框 + 红色错误卡片 |
| 付款码无效（后端验证失败） | 扫码框变红 + "付款码无效或已过期，请重新扫描" | 输入框红框 + "付款码无效或已过期" |
| 空输入 | — | 输入框红框 + "请输入付款码" |

所有错误状态 2.5~3 秒后自动恢复，允许重新操作。

### 11.8 支付确认数据结构（v1.1 更新）

`confirm` 事件除原有字段外，新增 `scannedCode` 字段：

```json
{
  "paymentMethod": "offline_wechat",
  "paymentMethodName": "微信",
  "amount": 10.00,
  "assetDeduction": { "prepaid": 60.00, "coins": 20, "external": 10.00 },
  "paymentLines": [
    { "type": "prepaid", "amount": 60.00 },
    { "type": "game_coin", "amount": 20, "exchange_rate": "1:1" },
    { "type": "wechat", "amount": 10.00 }
  ],
  "scannedCode": "280123456789012345"
}
```

`scannedCode` 在现金/会员资产抵扣时为 `null`。
```

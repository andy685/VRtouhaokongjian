# 退款逻辑说明

## 资金模型

```
用户充值 ¥100
    ↓
商家预存款账户 +¥100（全额到账，平台不抽成）
    ↓
商家向平台购买游戏豆 ¥50  →  平台收入 ¥50
                              ↓
                          代理商分润（推广游戏豆销售）
    ↓
商家提现 ¥50  →  扣除手续费 →  到账 ¥49.5（平台赚手续费）
```

**关键点：**
- 用户充值 = 全额归商家
- 平台收入 = 游戏豆销售 + 提现手续费
- 代理商分润 = 推广游戏豆销售，不抽充值流水

## 两种退款流程

### 1. 未结算订单 → 拉卡拉自动退款

```
订单状态：完成
     ↓
用户申请退款
     ↓
判断：该订单是否已结算给商家？
     ├── 否 → 调用拉卡拉退款 API
     |         ↓
     |      拉卡拉从待结算资金中扣款
     |         ↓
     |      原路退回给用户（微信/支付宝）
     |         ↓
     |      微信异步回调通知退款结果
     |         ↓
     |      订单状态：完成 → 退款中 → 已退款
     |
     └── 是 → 走线下退款流程
```

### 2. 已结算订单 → 商家线下退款

```
订单状态：完成（已结算）
     ↓
用户申请退款
     ↓
提示：该订单已结算，款项已到商家账户
     ↓
商家线下退给用户（现金/微信转账/支付宝转账）
     ↓
商家在系统确认退款 + 上传转账凭证（选填）
     ↓
订单状态：完成 → 已退款（线下）
```

## 接口设计

### 发起退款

```
POST /api/orders/{orderNo}/refund

请求参数：
{
  "refundReason": "客户不想要了",      // 必填
  "items": [
    { "name": "过山车VR", "qty": 1 }  // 退哪些商品、退多少
  ],
  "receiptImg": "https://..."         // 线下凭证（选填，已结算时使用）
}

响应：
{
  "refundNo": "RF20260507123456",     // 退款单号
  "refundAmount": 36.10,              // 退款金额
  "method": "lakala_auto",            // 退款方式：lakala_auto / offline
  "status": "processing"              // processing / completed
}
```

### 拉卡拉退款回调

```
POST /api/payment/lakala/refund-callback

{
  "outRefundNo": "RF20260507123456",
  "refundId": "微信退款单号",
  "refundStatus": "SUCCESS",           // SUCCESS / CLOSED / ABNORMAL
  "refundAmount": 3610,               // 单位：分
  "successTime": "2026-05-07T10:35:00+08:00"
}
```

## 退款原因枚举

| 原因 | 说明 |
|------|------|
| 客户不想要了 | 顾客改变主意 |
| 游戏体验异常 | 游戏卡顿、闪退等问题 |
| 操作错误 | 买错项目或数量 |
| 设备故障 | 头显/主机故障无法游玩 |
| 重复付款 | 同一笔订单付了两次 |
| 其他 | 自定义输入 |

---

## 风控规则

| 规则 | 说明 |
|------|------|
| **退款原因必填** | 每一笔退款必须选择或填写原因 |
| **操作人记录** | 记录谁操作的退款，关联员工账号 |
| **不可重复退** | 已退款的订单不能再次退款 |
| **已结算标记** | 已结算订单退款时必须有管理员确认 |
| **拉卡拉超时处理** | 退款请求发出后未收到回调，需人工核查 |

---

## 备注

- 退款不影响已有结算记录（已结算的订单退款后，该笔结算记录不会撤回，仅标记退款）
- 拉卡拉退款是异步的，需等待微信支付回调通知退款结果
- 线下退款由商家自行操作，系统仅做记录和标记

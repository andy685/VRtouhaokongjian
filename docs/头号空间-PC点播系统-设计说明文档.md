## 3. 页面结构总览

建议把设计稿中的页面组织为以下信息架构：

### 3.1 一级结构

| 一级区域 | 设计表现 | 说明 |
|----------|----------|------|
| 点播 | 顶部主导航默认高亮 | 面向顾客的主流程入口，承担选游戏、支付、分配设备、结算 |
| 商店 | 顶部导航可切换 | 更偏店员或内容安装视角，承载游戏详情与安装能力 |
| 管理 | 顶部导航可切换 | 进入设备/播控/参数配置等店员管理能力 |

### 3.2 顶部通用框架

设计稿中多个页面复用了统一顶栏，包含以下通用元素：

| 区域 | 内容 |
|------|------|
| 左侧 | 品牌 Logo / 门店标识 |
| 中部 | 一级导航：点播、商店、管理 |
| 右侧 | 游戏豆充值、消息提醒、下载/同步入口、关闭或返回 |
| 页面主体首屏 | 搜索框、分类筛选、游戏卡片区 |
| 页面底部操作区 | 分屏设置、设备列表、状态提示、门店名称 |

这说明点播系统不是一次性流程页，而是常驻 kiosk 式应用框架。

## 4. 页面树与功能说明

### 4.1 顾客主流程页面

### 4.1.1 主界面/游戏大厅

| 页面 | 关键元素 | 设计结论 |
|------|----------|----------|
| 主界面-默认态 | 搜索框、分类 Tab、推荐卡、游戏宫格、底部功能入口 | 顾客进入终端后的主操作页 |
| 主界面-搜索结果 | 搜索词回填、结果标题、命中卡片列表 | 支持按游戏名称快速定位 |
| 主界面-搜索空态 | “未找到”空状态插画与提示 | 需要为搜索失败提供独立空态 |
| 主界面-分屏 | 分屏设置弹层 + 单屏/双屏/四屏 | 运行中存在多画面监看或展示需求 |

设计要点：

- 分类在视觉上已明确为内容筛选主入口，而不是次级标签。
- 游戏卡片同时承载推荐角标、玩法标签、时长、封面信息。
- 底部存在“分屏设置”和“设备列表”两个快捷入口，说明主界面不仅用于选游戏，也承担设备态势入口。

### 4.1.2 游戏详情页

设计稿中的详情页已不是简单介绍页，而是启动决策页，建议拆为统一详情骨架 + 平台差异状态。

统一信息区包含：

- 游戏封面/视频预览区
- 游戏名称
- 价格或游戏豆消耗
- 标签：如惊悚恐怖、竞技射击、最新VR
- 简介文案
- 游戏参数：包体大小、时长、是否支持行走/射击、语言、外设支持、空间要求、版本、发布时间

平台差异动作区：

| 场景 | 页面表现 | 说明 |
|------|----------|------|
| 主机游戏 | CTA 偏向“安装到主机”或直接启动 | 游戏运行在主机侧 |
| 一体机游戏 | 需要先检查头显接入状态 | 游戏需安装到头显 |
| 特殊类型游戏 | 设计稿备注提示“需要提示启动播控系统” | 某些内容可能不走普通启动链路 |

### 4.1.3 启动/设备选择弹层

设计稿中“游戏启动”弹层已明确体现多人和设备分配逻辑：

| 组成 | 设计内容 |
|------|----------|
| 顶部图例 | 已占、未占、体验中、不可用 |
| 中部 | 设备卡片，展示设备名称与可选状态 |
| 底部左侧 | 已选游戏卡片、时长、单价 |
| 底部中部 | 已选人数、需支付金额 |
| 底部右侧 | 支付方式按钮：会员支付、微信支付、支付宝支付、员工点播 |

设计结论：

- 设备分配不是纯系统自动黑盒，用户或店员可感知设备可用性。
- 多人模式下需要先选设备，再结算。
- “员工点播”是单独入口，对所有人可见，但只有员工完成身份校验后可执行。

### 4.1.4 支付与结果态

从画布中的一组二维码与弹窗稿可以确认，支付至少存在以下状态：

| 状态 | 设计含义 |
|------|----------|
| 扫码展示态 | 展示二维码或反扫识别入口 |
| 支付中 | 等待支付回调或状态轮询 |
| 支付成功 | 结果确认，进入设备引导或返回 |
| 支付失败 | 失败提示与重试 |
| 取消/关闭 | 用户主动关闭支付弹层 |

结合现有 PRD，建议在合并时明确两类支付链路：

1. 顾客扫屏幕码支付
2. 终端/扫码枪反扫顾客付款码或会员码

当前设计稿对这两类链路都有体现，但需在正式需求中明确各自使用场景。

---

### 4.2 店员操作与设备管理页面

### 4.2.1 设备分屏与播控入口

设计稿中的“主界面-分屏”说明 PC 端除了顾客视角，还承担现场播控/监看职责。

| 功能 | 设计体现 | 推断说明 |
|------|----------|----------|
| 单屏/双屏/四屏 | 分屏设置弹层 | 用于展示或监控多个设备画面 |
| 设备列表 | 底部快捷入口 | 快速切换设备 |
| 播控快捷键 | 头显校正、投屏、结束播放、更多 | 运行中快捷操作集合 |
| 更多菜单 | 查找设备、音量调节、瞳距微调 | 单设备快捷控制菜单 |

这部分建议在 PRD 中独立成“运行中控制面板/播控工具条”，不要仅放在“设置”章节里。

### 4.2.2 头显管理

设计稿下方一整排页面与 PRD 4.3.6 高度一致，可确认以下页面集合真实存在：

| 页面类型 | 关键内容 |
|----------|----------|
| 设备列表页 | 已绑定设备列表、状态、搜索或切换 |
| 参数设置页 | 瞳距、刷新率、分辨率、亮度、音量限制、待机策略 |
| 安装游戏页 | 可安装游戏列表、已安装/可选状态、安装操作 |
| 批量应用页 | 预设方案或批量下发 |
| 保存反馈页 | 保存成功或同步成功状态 |

设计结论：

- 头显管理不是简单弹窗，而是一整套二级管理信息架构。
- 终端需要支持“设备对象切换”后展示不同配置详情。
- 参数页视觉上已经支持“卡片化选项 + 表单化参数”混合布局。

### 4.2.3 密码进入/店员模式

画布中存在多张店员授权相关页面，整体表现为：

- 先进入蓝色授权页
- 输入密码或验证码
- 验证通过后进入管理能力

建议在 PRD 合并时明确：

| 项目 | 建议规则 |
|------|----------|
| 入口 | 仅管理/设置入口触发，不影响顾客主流程 |
| 鉴权方式 | 店员密码为主，可扩展扫码授权 |
| 超时 | 管理模式无操作自动退出 |
| 退出后 | 回到顾客主界面 |

### 4.3 安装与内容分发流程页面

设计稿对“一体机安装游戏”给了比较完整的状态链，说明这部分不是后台能力，而是 PC 终端现场能力。

### 4.3.1 安装链路页面

| 页面/状态 | 设计表达 | 业务含义 |
|-----------|----------|----------|
| 下载前检测 | 先检查游戏属于主机还是一体机 | 安装动作依赖游戏运行平台 |
| 等待一体机接入 | 弹层提示 USB 3.0 连接 | 一体机安装需要物理接入 |
| 未连接数据线 | 禁用安装按钮 | 不满足安装前置条件 |
| 安装到主机 | 主机游戏的安装动作 | 与一体机安装分离 |
| 安装进度 | 进度条、传输状态 | 需展示实时反馈 |
| 安装成功 | 成功结果页 | 允许返回或继续操作 |

### 4.3.2 从设计稿可直接提炼的规则

| 规则 | 设计证据 | 建议并入 PRD |
|------|----------|--------------|
| 游戏平台决定安装动作 | 设计稿备注“检测到头显类型/主机类型的游戏” | 并入 `4.3.7 游戏运营设置兼容` |
| 一体机安装依赖有线连接 | 弹层明确写 `USB 3.0` | 并入头显安装规则 |
| 特殊类型游戏需播控提示 | 设计注释中直接说明 | 并入特殊内容启动规则 |
| 一台主机绑定一个点播系统 | 设计注释中直接说明 | 并入系统绑定模型 |

## 5. 功能模块拆解

建议将 PC 点播系统拆为以下 8 个模块，后续合并到 PRD 时也按这个粒度落章节或子章节。

### 5.1 内容浏览模块

- 游戏分类筛选
- 搜索与空态
- 推荐位展示
- 游戏卡片信息展示

### 5.2 游戏详情与购买决策模块

- 详情展示
- 平台/玩法/人数/标签展示
- 价格或游戏豆消耗展示
- 启动前校验

### 5.3 身份与支付模块

- 会员身份识别
- 微信/支付宝支付
- 会员资产抵扣
- 员工点播/内部核销
- 支付中/成功/失败/取消状态

### 5.4 设备分配模块

- 设备状态可视化
- 单人/多人设备选择
- 已选人数与金额联动
- 设备占用状态判断

### 5.5 运行中播控模块

- 分屏设置
- 设备列表
- 头显校正
- 投屏
- 结束播放
- 更多菜单中的查找设备、音量、瞳距操作

### 5.6 头显管理模块

- 设备绑定/解绑
- 设备信息查看
- 参数设置
- 预设方案
- 批量应用

### 5.7 内容安装模块

- 下载前检测
- 平台识别
- 连接检测
- 安装进度
- 安装结果反馈

### 5.8 店员授权与安全模块

- 密码进入管理模式
- 管理态自动退出
- 顾客态与管理态隔离

## 7. 与现有 PRD 的对照结论

### 7.1 已在 PRD 中有定义，设计稿进一步坐实的内容

- PC 终端承担完整交互，VR 头显不承载通用 UI。
- 头显管理确实包含设备信息、参数设置、安装游戏三块核心能力。
- 游戏运行平台需要区分主机游戏与一体机游戏。
- 多人模式需要设备分配与人数联动。

### 7.2 建议补充到 PRD 的内容

| 补充项 | 原因 |

| 搜索空态、搜索结果页说明 | 设计稿已成型，PRD 中未单列 |
| 设备状态图例定义 | 设计稿已给出完整状态色和状态名 |
| 员工点播入口定义 | 设计稿已出现，后续已在 PRD 中明确业务口径 |

| 特殊类型游戏需提示“启动播控系统” | 应形成特殊流程规则 |
